KeiSeiKit is organised in **four cooperating layers**. Each layer is
independently replaceable and only depends on the layer below — the
Constructor Pattern at the system scale.

## Four layers

### 1. API
The thin outer skin — CLI binaries, HTTP/JSON endpoints, and tool
manifests. The API translates user intent into a request the Brain can
plan against. No business logic lives here.

### 2. AI Brain
The planner. Given a request, the Brain selects a recipe (a TOML DAG)
or composes one on the fly. It owns the **14 Brain Tools** and chooses
between **5 Router types** (filesystem, http, queue, memory, agent).

### 3. Runtime Core
Recipe engine + scheduler. Walks the DAG, dispatches each step to one
of the 13 atomic primitives, persists artefacts, and emits structured
trace events. The Core is what makes a recipe reproducible across
machines.

### 4. Kernel
Rust crates and OS bindings — `kei-ledger`, `kei-tlog`, `kei-ping`,
`kei-disk-scan`, the hook-runner, the agent worktree manager. Pure
mechanism, no policy.

## The 13 atomic primitives

Every recipe decomposes into a sequence drawn from this set:

| Primitive  | Role                                       |
|------------|--------------------------------------------|
| INGEST     | Bring an external artefact into the runtime |
| PARSE      | Structure raw input into a typed record     |
| TRANSFORM  | Pure function over typed records            |
| ENRICH     | Augment with derived fields (LLM, lookup)   |
| VALIDATE   | Reject records that fail a schema           |
| DECIDE     | Branch on a condition                       |
| DISPATCH   | Route a record to a downstream sink         |
| PERSIST    | Write to durable storage                    |
| RETRIEVE   | Read from durable storage                   |
| SEQUENCE   | Order a stream                              |
| EMIT       | Publish an event                            |
| OBSERVE    | Subscribe to a stream                       |
| EXECUTE    | Run an external process                     |

<Aside type="note">
Per-primitive documentation is auto-generated by the `keidocs`
Rust runtime from the source crate's README and tests. See
[Primitives](/primitives/) for the live catalogue.
</Aside>

## Constructor Pattern at every scale

One file = one class = one responsibility. One layer = one concern.
One primitive = one verb. The same rule that keeps a 200-LOC file
readable keeps a 4-layer system explainable.
